## 2. 本轮目标

### 2.1 产品目标

- [x] 让 `应用排行 / 应用分布 / 总时长` 更接近真实参与时间
- [x] 让 `专注时间流` 更准确表达“这段时间主要在关注什么”
- [x] 减少 AI 等待、构建等待、前台挂起等场景的明显多记
- [x] 在不增加大量新设置项的前提下完成本轮收敛

### 2.2 核心原则

- [x] 时间流连续性不等于统计有效时长
- [x] 切到其他应用后的时间不能因为回到原应用就补回原应用统计
- [x] 设置页继续保持简单，用户只理解少量概念
- [x] tracking correctness owner 继续留在 Rust tracking 主链，而不是页面层

## 3. 最终产品口径

### 3.1 设置项

- [x] 将原 `时间流合并窗口` 升级为 `连续关注窗口`
- [x] 保留 `连续关注窗口` 的分钟滑杆形式
- [x] 滑杆范围继续保持 `1 - 5 分钟`
- [x] 默认值继续保持 `3 分钟`
- [x] 第一阶段继续保留 `自动挂机判定`
- [x] 保留 `专注时间流最少时长`
- [x] 保留 `暂停追踪`

### 3.2 设置文案

- [x] `自动挂机判定`
  说明文案统一为：
  `超过这个时间未操作时，会在最后一次输入时结束当前记录。`

- [x] `连续关注窗口`
  说明文案统一为：
  `短暂未操作，或暂时切到其他应用后在这个时间内返回，仍会视为同一轮连续关注。`

### 3.3 连续关注窗口的统一语义

- [x] `连续关注窗口` 是本轮唯一的短中断时间窗口
- [x] 它同时覆盖：
  - [x] 同一应用里的短暂静默
  - [x] 短暂切到其他应用后又回来
- [x] 第一阶段不再引入第二条并列的短中断时间滑杆
- [x] 第一阶段口径统一为：连续关注相关判断先以同一个分钟值为准

### 3.4 自动挂机判定的阶段性角色

- [x] 第一阶段保留 `自动挂机判定`
- [x] 第一阶段中，`自动挂机判定` 不再承担短中断连续性的主职责
- [x] 第一阶段中，短静默与短切出回流的主规则统一由 `连续关注窗口` 承担
- [x] 第一阶段中，`自动挂机判定` 只保留为长时间无操作时的兜底封口规则
- [ ] 第二阶段再决定 `自动挂机判定` 是否继续保留为独立设置

### 3.5 第二阶段预留命名

- [x] 第二阶段如引入低交互但真实参与场景的专用能力，产品标题统一使用 `持续参与时长`

## 4. 真实使用时间口径

### 4.1 本轮定义

- [x] 真实使用时间应尽量反映“用户确实在参与当前任务”的时间
- [x] 它不再粗暴等于“前台应用持续存在 + AFK 尚未触发”
- [x] 它也不能因为时间流连续展示就把整个跨度都算成有效时长

### 4.2 第一阶段口径

- [x] 第一阶段先用 `连续关注窗口` 统一控制短暂静默与短暂切出回流
- [x] 第一阶段优先修正“明显多记”
- [x] 超过 `连续关注窗口` 后，连续关注应视为结束
- [x] 超过 `连续关注窗口` 后，不能继续依赖 `自动挂机判定` 维持短中断连续性
- [x] `自动挂机判定` 仅保留为长时间无操作时的兜底封口

### 4.3 时间流与统计的强约束

- [x] 时间流合并只表达“同一轮连续关注”
- [x] 时间归属仍按实际发生时的应用来算
- [x] 回到原应用不等于中间时间归原应用
- [x] 不能出现“为了时间流好看而污染应用统计”的情况

## 5. 关键场景口径

### 5.1 编码 -> AI 等待 -> 再次编码

场景：
- 先交互 `4 分钟`
- AI 在前台运行 `5 分钟`
- 再次交互 `5 分钟`

本轮口径：
- [x] 不能继续把整段等待完整记入有效时间
- [x] 如果等待超过 `连续关注窗口`，时间流应拆段
- [x] 中间等待不能被自动补成完整有效使用时长

### 5.2 同一应用内的短暂停顿

场景：
- 输入后停下来读输出、看报错、思考
- 几十秒到数分钟内继续操作

本轮口径：
- [x] 不应切得过碎
- [x] 这类短停顿仍可视为同一轮连续关注

### 5.3 短暂切到别的应用又回来

场景：
- 从主应用短暂切到浏览器、聊天工具或小工具
- 很快返回原应用

本轮口径：
- [x] 时间流可以仍视为同一轮连续关注
- [x] 中间切出去的时间不能自动补回原应用
- [x] 如果中间应用存在真实交互，那段时间归中间应用

### 5.4 视频与会议

场景：
- 长时间看视频、直播、网课或开会，输入很少

本轮口径：
- [x] 第一阶段不承诺完整解决
- [x] 第一阶段只保留为后续增强方向
- [x] 第一阶段不为此把设置模型复杂化
- [x] 第二阶段如果纳入这类场景，统一使用 `持续参与时长` 命名
