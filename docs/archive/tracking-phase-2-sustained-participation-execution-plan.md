## 3. 第二阶段完成定义

第二阶段完成后，应满足以下产品结果：

- [x] `持续参与时长` 只服务视频 / 会议 / 课程 / 直播等低交互参与场景
- [x] 交互类应用不再因为 `持续参与时长` 被延长计时
- [x] 视频 / 会议类应用只有命中高置信持续参与信号时，才允许延长计时
- [x] 浏览器前台仅仅停留在视频网站或会议网页，不能自动等于持续参与
- [x] 超过 `持续参与时长` 后，会在阈值边界封口，而不是继续挂住旧 session
- [x] 前端状态文案、时间流、统计口径与 Rust 主链行为保持一致

如果这些条件没有同时成立，则第二阶段仍视为未完成。

## 4. 最终产品口径

### 4.1 连续关注窗口

- [x] `连续关注窗口` 只负责交互类应用的短中断连续性判断
- [ ] 它覆盖：
  - [x] 同一应用内短暂未操作
  - [x] 短暂切到其他应用后返回
- [x] 它不负责视频 / 会议类应用的长时间低交互延长

### 4.2 持续参与时长

- [x] `持续参与时长` 只负责低交互参与场景的延长判断
- [ ] 它适用于：
  - [x] 视频类客户端
  - [x] 会议类客户端
  - [x] 后续明确纳入的课程 / 直播客户端
- [ ] 它不应用于：
  - [x] IDE / 编辑器 / 设计工具
  - [x] 普通浏览器阅读 / 搜索 / 文档场景
  - [x] 普通聊天工具

### 4.3 第二阶段的关键收口原则

- [x] “允许持续参与”不等于“当前就一定在持续参与”
- [x] 只有命中持续参与信号，才允许继续计时
- [x] 没有持续参与信号时，会退回第一阶段口径
- [x] 没有为了弥补视频 / 会议漏记而重新放宽所有低交互场景

## 10. 暂停条件

出现以下任一情况时，第二阶段应暂停，不继续硬做：

- [ ] 为了做浏览器视频支持，不得不引入浏览器扩展或页面注入
- [ ] 为了做会议支持，不得不把平台 API 细节直接写回 `engine/*`
- [ ] 为了做低交互支持，不得不重新放宽所有应用的空闲容忍
- [ ] 无法获得高置信信号，却准备继续扩大 allowlist 范围
- [ ] 前端分类表或默认映射表开始反向成为 Rust tracking 的真实 owner
